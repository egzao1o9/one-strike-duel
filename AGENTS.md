# AGENTS.md

このファイルは、このリポジトリで作業するエージェント向けの最小ガイドです。  
実装・修正時は、まずこの方針に従ってください。

## 目的

このリポジトリは `One Strike Duel` の開発用です。  
現在は次の2本立てで進んでいます。

- `Python` 側
  - 参照実装
  - 大量シミュレーション
  - カードデータ生成
  - バランス検証
- `frontend/` 側
  - `React + TypeScript + Vite`
  - 人間が遊ぶ Web プロトタイプ
  - itch.io 向けブラウザ版の土台

今後の主戦場は **`frontend/`** です。  
ただし、カードデータや参照ルールは Python 側も引き続き重要です。

## 基本方針

### 1. 機能実装を優先する

現段階では、見た目の作り込みよりも

- ゲームループが最後まで自然に回ること
- 効果処理が検証できること
- デバッグしやすいこと

を優先します。

### 2. 日本語を優先する

説明文、ドキュメント、UI文言はできるだけ日本語で扱ってください。  
英語はライブラリや型名など、コード上自然なものに限ります。

### 3. Python 完全一致は必須ではない

Web 版は、Python 参照実装を参考にしつつ進めます。  
ただし、今後は **Web での遊びやすさや実装の自然さ** を優先してよく、必ずしも Python 完全一致である必要はありません。

### 4. デバッグ機能を重視する

このゲームは効果順、対象選択、公開順が重要です。  
そのため、以下は積極的に整備してください。

- 任意盤面を作るデバッグ機能
- 特定効果だけ試す導線
- ログ表示
- 検証用の Debug カード

## このゲームの実装上の重要ポイント

### 公開情報と非公開情報を分ける

このゲームは秘匿情報が重要です。  
実装では常に以下を意識してください。

- プレイヤー自身が見えてよい情報
- 相手に見えてよい情報
- 完全状態

### 公開順が重要

バトルのカードは **1枚ずつ公開** され、公開順に処理されます。  
今後も以下のような効果が増える前提です。

- 自分の公開済みカードを対象にする
- 相手の公開済みカードを対象にする
- 公開時に発動する
- セット時に割り込む

そのため、効果実装時は

- どのタイミングか
- どの対象か
- 何が公開済みか

を明確に扱ってください。

### 対象選択系は汎用化する

今後想定している対象の例:

- 自分のセットカード
- 自分の公開済み battle カード
- 相手のセットカード
- 相手の公開済みカード
- 山札上位 n 枚
- 捨て札
- 使用済み

このため、対象選択UIや payload は都度ハードコードせず、できるだけ共通化してください。

## カードデータの扱い

編集元は `data/cards_src/` です。  
`data/cards.json` は生成物です。

重要なテキスト項目:

- `effect_text`
  - ルール上の効果説明
- `flavor_text`
  - フレーバー
- `notes`
  - 開発用メモ
  - UI には基本表示しない

### カード生成関連コマンド

```bash
python -m sim.validate_cards
python -m sim.refresh_cards
```

### CSV 入出力

```bash
python -m sim.export_cards_csv
python -m sim.import_cards_csv --cards-input data/cards_export.csv --effects-input data/card_effects_export.csv
```

## frontend の扱い

Web プロトタイプは `frontend/` にあります。

### 起動

```bash
cd frontend
npm install
npm run dev
```

### 確認

```bash
cd frontend
npm run typecheck
npm run build
```

実装変更後は、少なくとも `typecheck` と `build` を通してください。

## 現在の優先順位

おおむね以下を優先します。

1. Web 版のメインループ完成
2. 効果処理の拡張
3. デバッグ体験の改善
4. CPU 戦の品質向上
5. UI の磨き込み

  の流れを自然にする

- 対象選択系を今後増やしやすくする

## やってよいこと / 注意点

### やってよいこと

- Debug カードの追加
- Debug UI の追加
- 効果選択UIの汎用化
- 一時的な検証用実装

### 注意点

- 既存 UI の大きなレイアウト変更は、ユーザーの指示に沿って慎重に行う
- `notes` を表示用に使わない
- 同名カードがある前提で、選択は index や instance 単位で扱う
- battle の処理は「全部同時」ではなく、公開順・割り込みを意識する

## 禁止事項 / 勝手にやらないこと

- 指示されていない大規模リファクタをしない
- 指示されていないカードバランス調整をしない
- 指示されていないドラフトルール変更をしない
- `data/cards.json` を直接編集しない
- 秘匿情報を相手UIに表示しない
- 既存のログ形式・保存形式を勝手に破壊しない
- 見た目の大幅変更は指示がある時だけ行う

## Stop Conditions

以下の場合は、勝手に進めずに停止して確認してください。

- 仕様が曖昧で、ゲーム性に大きく影響する場合
- 秘匿情報の扱いが不明な場合
- 既存データ形式やログ形式の破壊が必要になりそうな場合
- 大規模な設計変更が必要になりそうな場合
- Python参照実装とWeb版のどちらを優先すべきか判断が必要な場合
- UIの見た目や操作感に大きな判断が必要な場合
