# Bot挙動リファレンス

## 目的

この文書は、現時点の標準系 Bot が何を重視しているかをまとめたものです。

- 対戦中の判断基準
- ドラフト時の判断基準
- `Standard / Aggro / Guard` の違い

対象コード:

- `bots/public_info_bot.py`
- `engine/drafting.py`
- `engine/analysis.py`

## 対戦Botの全体像

標準系の対戦Botは `PublicInfoBot` 系です。

- `StandardBot`
- `AggroBot`
- `GuardBot`
- `ControlBot`
- `BlessingBot`

この3種は判断ロジック自体は同じで、重みだけが違います。

### 性格の違い

`StandardBot`

- 基準型
- 攻撃、防御、速度、control をほぼ均等に見る
- 極端な押し引きはしにくい

`AggroBot`

- 攻撃寄り
- `attack_weight` と `speed_weight` が高い
- `risk_penalty` が低く、カード消費をあまり怖がらない
- 継続攻撃も見やすい

`GuardBot`

- 防御寄り
- `block_weight` と `control_weight` が高い
- `risk_penalty` が高く、無駄打ちを嫌う
- 防いで次ターンに回す判断が強い

`ControlBot`

- control 寄り
- 公開情報、手札公開、ドロー、盤面干渉を厚く見る
- battle 一辺倒より、control を battle に混ぜた小技を許容する

`BlessingBot`

- blessing 寄り
- blessing 本体と blessing 支援 control を強く見る
- 加護を回しながら、最低限の battle でつなぐ方針

## 対戦中に見ている情報

### 自分側

- 現在の手札
- 自分の公開デッキ
- 自分の非公開デッキ
- 自分の伏せ枚数
- 自分の現在の伏せカード
- 自分の control / blessing 状態
- 手札から作れる
  - 最大攻撃ライン
  - 最大防御ライン
  - 最大速度ライン

### 相手側

- 相手の公開デッキ
- 相手の公開ドラフト履歴
- 相手の捨て札
- 相手の使用済みカード
- 相手の手札枚数
- 相手の山札枚数
- 相手の伏せ枚数
- 相手がすでに `pass` したか
- 相手の control / blessing 状態

### 推定しているもの

- 相手の公開札から見た平均的な
  - 攻撃
  - 防御
  - 速度
- カードプール全体から見た相手の非公開候補
- 公開ドラフトで何を優先して取ったかに基づく、非公開候補の重み補正
- 相手の最終的な攻撃ライン、防御ライン、速度ライン

## マリガンで重視していること

### 基本方針

手札を1枚ずつ点数化して、弱い札から返します。

### battle カードの評価

- 攻撃が高いほど加点
- 防御が高いほど加点
- 速度が高いほど加点
- マイナス値は減点
- 効果付きは加点
- 手札内のベスト攻撃コンボ、ベスト防御コンボ、ベスト速度コンボに含まれる札は加点

### control / blessing の評価

- 効果の強さ
- 今の手札との相性
- 相手公開情報への刺さり
- 既に blessing が置かれているか

### 現状の性格

- `AggroBot` は低打点札を返しやすい
- `GuardBot` は受けに役立たない札を返しやすい
- ただし完全な「手札全体プラン」ではなく、まだカード単位評価の比重が高いです

## コントロールフェーズで重視していること

`choose_control_card()` では、出せる `control` と `blessing` を点数化して、一番高いものを選びます。

### 共通で強く見るもの

- `draw`
- 相手手札や山札の公開
- 相手防御低下
- 自分速度上昇
- 自分防御上昇
- 相手攻撃低下
- 相手1枚目無効化
- 追加ドロー系ルール変更

### 個別カードの追加評価

一部のカードはIDごとに追加補正があります。

例:

- 相手 blessing を裏返す
- 相手 blessing を捨てる
- 相手 blessing を封じる
- 裏向き blessing を捨てる
- 手札入れ替え
- オープニング公開系

### blessing の扱い

- すでに自分の blessing があると評価を下げる
- 強い blessing 破壊札や封印札は、相手 blessing が見えていると強くなる

## バトルフェーズで重視していること

### まずモードを決める

各ターンのバトル判断は最初にモードを選びます。

- `attack`
- `defense`
- `probe`

### Probe Mode

様子見です。

- 初手の空盤面は基本ここに入る
- `set` を強く推す
- `set_pass` と `pass` は強く下げる
- 速度や防御を少し確保しつつ、次の判断材料を増やす

### Attack Mode

このターンで通しにいくモードです。

重視するもの:

- 相手防御を攻撃で抜けるか
- 速度勝ちできるか
- 最終的に相手の追加込みでも通せるか
- 追加で使うカードの価値が高すぎないか
- 次ターンの継続攻撃余地があるか

### Defense Mode

このターンは勝たなくてもよい、というモードです。

重視するもの:

- 相手攻撃を止められるか
- 過剰防御になっていないか
- 使う枚数が少ないか
- 強カードを温存できるか
- 次ターンに攻め返せるか

### set / set_pass / pass の考え方

`set`

- 基本行動
- 初手は特にこれを優先
- 相手より `+1枚` 置ける局面は加点

`set_pass`

- 勝負を閉じる宣言に近い扱い
- 初手では大きく下げる
- 相手の最後の合法追加込みで見ても
  - 勝てる
  - 守れる
  なら加点
- 2枚以下での早すぎる `set_pass` は強く下げる

`pass`

- すでに十分守れている時は加点
- 相手の最後の追加込みで危ないなら大きく下げる
- 早すぎる `pass` はかなり嫌う

## 対戦中の推定ロジック

相手の最終ライン推定は、単純平均ではありません。

見ているもの:

- 相手公開札の残り
- 相手の捨て札
- 相手の使用済み札
- 自分の持っている札
- カードプール残数
- 相手の公開ドラフト履歴

これらから、相手の非公開候補に重みを付けています。

### 何を補正しているか

- 公開ドラフトで攻撃札を優先していたなら、非公開側でも同系統をやや持っていると見る
- すでに使われた強札は、未知候補から差し引く
- 公開札で見えているものは、未知の hidden 候補から除く

## 予測ログで記録しているもの

各 battle 行動ごとに以下を記録しています。

- 予測した相手タイプ
  - `attack / defense / speed / balanced / none`
- 予測した相手最終ライン
- 相手の最終追加込みの予測ライン
- その予測に対する自分の対応方針
  - `probe`
  - `minimal_guard`
  - `full_guard`
  - `guard_and_counter`
  - `speed_race`
  - `attack_push`
  - `pressure_with_cover`

集計では以下が見えます。

- 予測命中率
- 予測タイプ別命中率
- 方針別成功率
- 攻撃、防御、速度の平均誤差

## ドラフトBotの全体像

標準系ドラフトBotは `PublicInfoDraftBot` 系です。

- `StandardDraftBot`
- `AggroDraftBot`
- `GuardDraftBot`
- `ControlDraftBot`
- `BlessingDraftBot`

これも対戦Botと同じく、ロジックは共通で重みだけ違います。

### 性格の違い

`StandardDraftBot`

- バランス型
- 公開/非公開の価値を平均的に見る

`AggroDraftBot`

- 攻撃札と hidden フィニッシャーを高く見る
- 公開カウンター性能は少し軽め
- reveal される不利をあまり嫌わない

`GuardDraftBot`

- 防御札、control、公開カウンター性能を高く見る
- hidden フィニッシャーは控えめ
- 公開で重い攻撃札を見せる不利を強めに嫌う

`ControlDraftBot`

- `control` を多めに取りにいく
- 公開情報操作、ドロー、干渉札を厚く見る
- battle は最低限の汎用札に寄せる

`BlessingDraftBot`

- blessing を他よりやや多めに見る
- blessing を捨てる、裏返す、封じる支援札も高く見る
- ただし取りすぎはまだペナルティ対象

## ドラフトで重視していること

### 1. カード単体点

`base_card_score()` が基礎です。

- rarity
- 攻撃
- 防御
- 速度
- マイナス値
- 効果数

### 2. デッキ不足分

`need_bonus()` で、残りピック数に対して不足している部分を埋めにいきます。

見るもの:

- `battle / control / blessing` の枚数
- `attack / speed / defense / control` の役割色
- `common / uncommon / rare` の最終配分

### 3. 重複ペナルティ

同じカードを多く取りすぎると減点します。

### 4. blessing 調整

`blessing_pick_adjustment()` で加護の取りすぎを防ぎます。

- 1枚目は加点
- 2枚目は少しだけ許容
- 3枚目以降は大きく減点
- blessing を捨てたり触れたりできる support があれば少し緩和

### 5. 公開/非公開の違い

`hidden`

- 高打点や奇襲札をやや高く見る
- フィニッシャーを隠して持つ価値を加点

`public`

- 相手公開デッキに対して受け札やカウンター札を高く見る
- 攻撃だけ高くて防御ゼロに近い札は少し下げる
- 相手に渡したくない札を取る、という denial も少し見る

## ドラフトでの重み差

### AggroDraftBot

強く見るもの:

- 攻撃
- 速度
- hidden フィニッシャー

軽く見るもの:

- 防御
- control
- 公開 denial

### GuardDraftBot

強く見るもの:

- 防御
- control
- 公開カウンター性能
- denial

軽く見るもの:

- hidden フィニッシャー

## 現状の弱点

### 対戦Bot

- 攻撃読みと速度読みの命中率はまだ低い
- 手札全体プランより、まだ単体札評価がやや強い
- control の高度な順序価値はまだ浅い

### ドラフトBot

- パック単位の相乗効果評価はまだ浅い
- blessing と blessing support の組み合わせ評価は最小限
- 将来のプレイ精度を見越した「プレイしやすさ」の評価は弱い

## 今後、特徴を増やすなら

追加しやすい軸は次です。

- 速度読み特化
- blessing 特化
- 少枚数勝負特化
- control 先置き特化
- 公開情報カウンター特化
- 長期戦リソース特化

今の `Standard / Aggro / Guard` は、これらの土台として使える状態です。
