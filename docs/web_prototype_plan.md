# Webプロトタイプ計画

## 目的

- `itch.io` に公開できるブラウザ版プロトタイプを作る。
- 人間が CPU 戦を通して、ゲームとして面白いかを検証できる状態にする。
- 既存の Python 実装を参照実装として活かしつつ、ゲーム本体は `TypeScript` へ移植する。

## 方針

- 配布形態は `ブラウザ版(Web)` とする。
- 最初の公開版は `1人用CPU戦` のみとする。
- ドラフトは `フルドラフト` を実装する。
- 対戦Botは最初は `Standard` のみ使う。
- UIは仮UIでよい。見た目より情報の正確さを優先する。
- 対戦ログは最初は文章ログでよい。
- ただし将来のリプレイ再生を見据えて、ログは `アクション単位` で持つ。

## 初回公開版の範囲

### 入れるもの

- タイトル画面
- フルドラフト画面
- バトル画面
- 結果画面

### 入れないもの

- 高品質なビジュアル演出
- チュートリアル

## 技術方針

### フロントエンド

- `React`
- `TypeScript`
- `Vite`

### 状態管理

- 初期は軽量でよい。
- 第一候補は `useReducer`
- 状態が煩雑になったら `Zustand` を検討する。

### ロジック実装

- ゲーム本体は `TypeScript` に移植する。
- Python は本番ランタイムでは使わない。
- Python は以下の用途で残す。
  - 大量試行シミュレーション
  - Bot 検証
  - TypeScript 実装の参照
  - ルール差分検出

## TypeScript移植の基本方針

- Python の構造をそのまま雑に写すのではなく、UI と接続しやすい形で整理する。
- ただしルール差異を避けるため、初期段階では Python の責務分割を大きく崩さない。

### 維持したい概念

- `GameState`
- `PlayerState`
- `PlayerView`
- `Card`
- `Effect`
- `BattleAction`
- `DraftEvent`
- `MatchLog`

### 重要な設計原則

- 秘密情報を含む完全状態と、プレイヤーが見えている公開状態を分ける。
- UI は常に `PlayerView` 相当の情報を読む。
- ログは結果テキストではなく `イベント列` を正として持つ。
- リプレイはそのイベント列を順送りして再生できる形を目指す。

## ログ方針

- 初期は `文章ログ` を表示する。
- 内部データは `イベント列` で持つ。
- 1イベントごとに次を持つ。
  - 種別
  - 実行プレイヤー
  - 公開情報への影響
  - battle への影響
  - 表示用テキスト

### 将来拡張

- アクション送りリプレイ
- 自動再生
- UIアニメーション再生
- 対戦共有

## テスト方針

- Python 実装を参照実装とみなす。
- TypeScript 側で同じ入力から同じ結果が出るかを確認する。

### 優先確認項目

- ドラフト結果の整合性
- 公開 / 非公開情報の扱い
- battle 解決順
- blessing の残留と裏返り
- control の持続
- 勝敗判定
