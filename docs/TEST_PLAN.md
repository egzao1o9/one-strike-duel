# Test Plan

## Parry / Retreat

### 自動確認

- `frontend`
  - `npm run typecheck`
  - `npm run build`

### 手動確認

#### パリィ

1. デバッグ対戦を開始する。
2. 片側に `Attack 2` 相当、もう片側に `Block 4` 相当の盤面を作る。
3. `Reveal` を進める。
4. 期待結果:
   - パリィが発生する。
   - 攻撃側にそのバトル中だけ `Block -2` が入る。
   - ログにパリィ発生が残る。
   - 次バトル開始時に、パリィされた側へ伏せ枚数制限がかかる。

#### パリィ非発生

1. `Attack 3` と `Block 4` の盤面を作る。
2. `Reveal` を進める。
3. 期待結果:
   - 攻撃は防がれるが、パリィは発生しない。

#### Attack 0 はパリィ対象外

1. `Attack 0` と `Block 5` の盤面を作る。
2. `Reveal` を進める。
3. 期待結果:
   - パリィは発生しない。

#### 次バトル制限

1. 一度パリィを発生させる。
2. 次のバトルで、制限を受けた側の伏せ選択を試す。
3. 期待結果:
   - 相手より多く伏せる行動は取れない。
   - バトル終了後、制限は解除される。

#### 両者制限相殺

1. 両者が次バトル制限を持つ状態をデバッグで作る。
2. 次バトルを開始する。
3. 期待結果:
   - 制限は相殺され、通常の伏せルールを使う。
   - ログに相殺が分かる情報が残る。

#### 撤退

1. 両者が伏せカードを 1 枚以上持つ盤面を作る。
2. どちらかが `retreat` を選ぶ。
3. 期待結果:
   - 撤退側の伏せカードはすべて捨て札へ移動する。
   - 相手は 1 枚だけ手札へ戻せる。
   - 相手の残り伏せカードは捨て札へ移動する。
   - バトル結果は `no_decision` になる。
   - 次ターンへ進む。

#### 伏せ 0 枚で撤退不可

1. 伏せカードが 0 枚の状態で `battle_select` に入る。
2. 期待結果:
   - `retreat` は選べない。

## 人間テストで見る観点

- 攻撃がパリィされることで、雑な総攻撃が減るか。
- 防御読みの成功体験が増えるか。
- `Block -2` が強すぎて逆転されすぎないか。
- 次バトル制限が重すぎて停滞を生まないか。
- `retreat` が便利すぎず、しかし選択肢として機能するか。
- ログと中央表示だけで、何が起きたか追えるか。
